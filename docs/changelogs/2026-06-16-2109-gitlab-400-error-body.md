# GitLab 400 error swallowed the response body

**Task:** Debug `Job 6 failed: Error: gitlab 400` reported with no detail in
logs (alongside unrelated `No onPostToolUseHook` ACP stderr noise).

## Root cause

`GitlabAdapter.openMergeRequest` (`packages/worker/src/adapters/gitlab.ts:44`)
threw `new Error(\`gitlab ${r.status}\`)` on a non-OK response without ever
reading the response body. GitLab puts the actual validation reason (invalid
branch, duplicate open MR, etc.) in the JSON body of a 400 — that information
was discarded before `withRetry` (which just rethrows `lastError` verbatim)
and `runStep` (which logs `err.message` faithfully) ever saw it. So every
downstream log/event/Telegram message could only ever say `"gitlab 400"`.

The `No onPostToolUseHook` lines are unrelated — a known harmless upstream
ACP stderr warning from `@zed-industries/claude-code-acp`, already covered by
the request-timeout fix in the previous session.

## Fix

`packages/worker/src/adapters/gitlab.ts`: read `r.text()` on failure and
append it to the thrown error message, so the real GitLab error reaches the
job's `step_error` event and Telegram report.
